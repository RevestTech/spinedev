Critical & High Security Findings
Critical Exposed Gmail App Password in .env File .env:30 1.0

GMAIL_APP_PASSWORD=uvhd...hkid is hardcoded in the committed .env file. Gmail app passwords grant full email account access — attackers can send phishing emails, exfiltrate data, and impersonate the organization.

Fix: Immediately revoke this app password from Google Account settings. Move .env to .gitignore. Store credentials in encrypted secret management (Vault, AWS Secrets Manager). Audit the email account for unauthorized access.
Critical Weak Default Certificate Password for E-Signatures src/services/esignature/certificate-manager.ts:57 1.0

Default hardcoded password 'fc-parser-dev-cert-2026' used for PKCS#12 signing certificates. If the .p12 file is obtained, this predictable password allows private key extraction and signature forgery.

Fix: Remove hardcoded default. Require CERT_PASSWORD in production via env validation. Generate strong random passwords with crypto.randomBytes(32). Regenerate existing certificates.
High Weak PKCS#12 Encryption (3DES) src/services/esignature/certificate-manager.ts:121 0.95

Certificate P12 encryption uses '3des' (Triple DES), which is deprecated with a 64-bit effective keyspace. Not considered secure for protecting private keys.

Fix: Change algorithm parameter from '3des' to 'aes256' in the forge.pkcs12.toPkcs12Asn1() call.
High Auth Bypass Flag Enabled in Development .env .env:3 1.0

ALLOW_AUTH_BYPASS=true allows requests with arbitrary x-user-email and x-user-role headers. If this .env leaks to staging/production, authentication is completely bypassed.

Fix: Create separate .env.development, .env.staging, .env.production files. Add fatal-exit guard in code when NODE_ENV=production && ALLOW_AUTH_BYPASS=true.
High Self-Signed Certificates for E-Signatures src/services/esignature/certificate-manager.ts:109-110 0.9

All signing certificates are self-signed with no chain of trust. Recipients cannot verify organizational identity. Legal enforceability of digital signatures is compromised.

Fix: Require CA-signed certificates in production via ESIGN_CERT_PATH/ESIGN_KEY_PATH env vars (already defined in env.ts). Fall back to self-signed only in development.
High Default Dangerous Passwords in Database URLs .env:11,16,19 1.0

Connection URLs contain change_me_in_production as default passwords for PostgreSQL and Redis. While env.ts validates against this in production, CI/CD pipelines must enforce rotation.

Fix: Validation already exists (good). Ensure CI/CD rejects default credentials. Use Docker secrets or Vault for production injection.

Critical Quality Findings
Critical Unsafe Type Casting with as any in Core Data Layer src/repositories/base.repository.ts (8 locations) 1.0

Base repository and multiple API routes use as any casts, bypassing TypeScript's type safety at the data boundary layer. API routes frequently initialize const updateData: any = {}.

Fix: Create typed builder patterns for update objects. Document intentional casts in base.repository.ts. Enable noImplicitAny: true in tsconfig.
Critical Untyped External API Responses (Salesforce, Google Drive) src/services/contracts/integration-hub.ts (11 locations) 0.95

All external API responses cast to as any without runtime schema validation. Malformed responses from Salesforce/Google Drive can crash downstream services silently.

Fix: Create Zod schemas for all external API responses. Validate before use: SalesforceTokenSchema.parse(await res.json()).
Critical Silent Error Swallowing in Document Processing Pipeline src/api/processing/routes.ts:267-271 0.98

Five critical processing steps (classification, section detection, entity harvesting, field extraction, validation) use catch (_e) { /* skip */ }. Failures are completely hidden — downstream systems receive incomplete data without notification.

Fix: Replace silent catches with structured error logging, dead-letter queuing, circuit breakers, and partial-result degradation. Each step should emit failure events.

Critical & High Performance Findings
Critical N+1 Query Loop in Entity Harvesting (100 sequential DB calls) src/services/intelligence/entity-harvester.ts:329-342 0.95

Loop over detected entities (up to 50) makes sequential database calls for each: resolveOrCreate() + linkEntityToDocument() = 100 queries per document instead of 2-3 batch operations.

Impact: +5-15 seconds latency per document processing.

Fix: Batch all entity resolution/linking using Promise.all() with concurrency control and PostgreSQL UPSERT patterns.
Critical Duplicate OCR Calls on Same Image (2x Processing Time) src/services/intelligence/ocr-engine.ts:44-54 0.98

Tesseract called twice on the same image buffer — once for plain text, once for TSV data. On 1,000-page documents, this adds 1-3 hours of redundant processing.

Impact: 100-200% increase in OCR processing time.

Fix: Use single Tesseract API call with output format parameter to get both text and TSV in one pass.
High Missing Index on entityAliases for Fuzzy Matching src/db/schema.ts:300-315 0.92

Entity resolution uses ilike(entityAliases.aliasName, pattern) with no trigram index. Full table scan on every lookup.

Impact: 50-200ms per entity lookup. With 100K+ aliases, queries take 500ms+.

Fix: Add trigram GiST index: CREATE INDEX idx_entity_aliases_name_trgm ON entity_aliases USING gist(alias_name gist_trgm_ops)
High Sequential Awaits in Entity Detection (Ollama Health Check) src/services/intelligence/entity-harvester.ts:295-299 0.88

Ollama health check and entity extraction run sequentially. Even if unhealthy, adds 5-10 seconds network latency per document.

Fix: Parallelize with timeout: Promise.race([ollama.extractEntities(...), timeout(5000)])
High Memory Accumulation in Entity Detection Map src/services/intelligence/entity-harvester.ts:178-290 0.82

All detected entities stored in Map<string, DetectedEntity> before filtering. 500+ page documents accumulate thousands of temporary entries — 50-200MB peak per document.

Fix: Process entities in streaming fashion. Discard non-matching entries immediately instead of storing all.
High Synchronous File Processing in Catalog Worker (10-20x Slower) src/api/catalog/routes.ts:58-139 0.9

File loop serializes metadata extraction and cataloging. 1,000 PDFs processed serially at 100ms each = 100 seconds. Could be 5-10 seconds with concurrency.

Fix: Use Promise.all() with concurrency control (pLimit with concurrency: 10).
