Spine recording layer

A Postgres database that records every interaction the multi-agent system has, so you can answer questions like "which agent cost us the most this week?" and "why did this directive fail?" without grepping log files.

What Spine is

Spine is a file-based multi-agent orchestration template. The agents communicate by writing markdown files on disk; small bash daemons poll those files and invoke a CLI agent (Claude Code, cursor-agent, aider, etc.) to do the actual work. One developer, one laptop, no SaaS orchestrator — just a shared directory and a handful of long-running shell scripts.

That filesystem bus is perfect for the everyday case. It falls over the moment you ask a real reporting question. "How many invocations did the planner do this month, and what did they cost?" turns into a grep across thirteen team folders and a calculator. The recording layer fixes that: Postgres sits next to the files and stores every interaction, so the questions become SQL queries and the dashboards become a single web page.

What we built

  1. The schema

    23 tables in Postgres 16 covering teams, workers, roles, assignments, directives, reports, cost rows, lifecycle events, and lookup tables for tiers, models, and providers. Migrations live in db/flyway/sql/ and run idempotently — V1 (core), V3 (multi-host columns), V4 (views), V5 (invocation durations), plus repeatable R__1 and R__2 for seed data.

  2. The outbox + watcher

    When a daemon finishes a job it appends a single JSON line to state/outbox.jsonl alongside its existing CSV cost record. A small Python watcher running in Docker reads each role's outbox, inserts the data into Postgres, and persists a byte-offset cursor so a crash never causes a replay or a gap.

  3. Lifecycle events

    Every interesting moment of an invocation — pickup, start, report-written, plan-written, reaped, awaiting-approval, approved — is recorded as an event row with a structured JSON payload. This becomes the append-only audit log: anything that happened, in the order it happened, queryable forever.

  4. Token-level cost capture

    When the executor is Claude Code, the daemon parses the JSON envelope it emits and stores input tokens, output tokens, cost in USD, and the model name on each invocation. That single piece of data multiplies the dashboard's usefulness — without it, cost is a single bill at the end of the month; with it, you can blame a specific role for a specific spike on a specific day.

  5. Views and SQL helpers

    Six database views aggregate the raw rows into the exact queries the dashboard needs: cost by role per day, cost by model, active workers, recent events, cost by outcome, and invocation durations. The team.sh budget-db and team.sh events-db shell commands query the same views from the terminal, so the dashboard and the CLI tell the same story.

  6. The dashboard

    What you're looking at. A small stdlib Python HTTP server runs every view and returns JSON; the page auto-refreshes every 5 seconds and updates rows in place with smooth animations. Filter by role, terminal type, and time range. Filters persist to localStorage and round-trip via the URL query string for shareable links.

Spine Hub Pillar 2 — code distribution

The first hub pillar is telemetry (heartbeats, cost projection, the fleet view). The second pillar is code distribution. Spine is a templating project — every installation consumes it. Pillar 2 lets an admin promote a specific commit of the SpineDevelopment template to a release channel (stable, beta, or canary), and every fleet member's updater.sh daemon fast-forwards its local template clone to that commit on the next tick.

The updater is opt-in. Enable it on a laptop with SPINE_UPDATE_ENABLED=1 plus SPINE_UPDATE_TEMPLATE_DIR=<path-to-clone> before running team.sh up. Per-instance drift status (current / drifted / unversioned / unknown_channel) is exposed on the Versions page, fed from the v_instance_drift view (V12).

Safety: the updater only git fetch + fast-forwards the template clone. It does not replace files in the consuming project's scripts/ tree, and it does not auto-restart running daemons — mid-engagement restarts are deliberately out of scope.

Why this matters

Without the recording layer, every observation about the agent team is grep-and-pray. With it, you can answer cost questions in SQL, ship dashboards in JS, and eventually swap the file bus for a real central-server mode without touching the schema. The schema is portable; the laptop runtime is not.

What's intentionally not done yet

  • Tenant scoping (V6 migration). A tenant_id column on every table for multi-customer deployments.
  • Token capture for cursor-agent and aider. Their output formats need bespoke parsers; today only Claude Code populates token and cost data.
  • Watchdog using PID files instead of pgrep -f. Needed before central-server mode where command lines can collide across hosts.
  • LISTEN/NOTIFY push semantics. Current daemons poll the filesystem every 8 seconds; central mode wants real Postgres pub/sub so a workspace event reaches every interested watcher instantly.
