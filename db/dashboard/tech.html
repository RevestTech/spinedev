How it works

The data path from agent invocation to dashboard pixel, end-to-end.

Data flow

Step by step

  1. Agent CLI (Claude Code / cursor-agent / aider)

    The actual AI model invocation. Reads a prompt via stdin, writes its response to stdout (and back to the directive file). Wall clock, exit code, and — for Claude Code — a JSON envelope with token usage are captured by the daemon that launched it.

  2. Bash daemon (scripts/team-agent-daemon.sh)

    One daemon per role per slot. It polls a markdown file every 8 seconds. When it picks up a fresh # Directive, it writes the prompt to a temp file (so the content never leaks via argv), invokes the agent through executor.sh, and records the outcome.

  3. Cost CSV + outbox JSONL (teams/<role>/state/)

    The daemon writes two things on every invocation: costs.csv (the original source of truth, 9 columns including outcome) and outbox.jsonl (the new data path — one JSON line per cost row plus one per lifecycle event). The outbox uses flock on Linux and atomic mkdir on macOS to handle concurrent appends safely.

  4. Python watcher (db/watcher/spine_watcher.py)

    Runs in Docker via make watch. Mounts the team workspaces read-only. Every 5 seconds, opens each role's outbox, seeks to the byte offset stored in spine_cursors/<role>.cursor, and inserts any new lines into Postgres in single-row transactions. The cursor only advances after a successful insert, so a crash never loses data and a restart never replays it.

  5. Postgres 16 (db/docker-compose.yml)

    Bound to host port 33000, schema in public. Flyway 10 manages migrations. Deterministic UUIDs (uuid5 of a stable namespace) so multiple watchers writing the same logical row converge instead of conflict. ENUM types for status columns, JSONB for payloads, TIMESTAMPTZ for every timestamp, and partial unique indexes for invariants like "one active assignment per worker".

  6. Views (V4__views.sql, V5__invocation_durations.sql)

    Six views aggregate the raw rows into what the UI wants: v_cost_by_role_day, v_cost_by_model, v_active_workers, v_recent_events, v_cost_by_outcome, and v_invocation_durations (with a LATERAL join that pairs every InvocationStarted event with its earliest terminal event for accurate duration math).

  7. Dashboard (db/dashboard/serve.py + index.html)

    A stdlib Python HTTP server with one dynamic endpoint (/pg-snapshot.json) that runs every view and returns the combined JSON. A 2-second in-memory cache stops multiple browser tabs from hammering Postgres. The HTML polls every 5 seconds, reconciles changed rows with smooth animations, and persists filter state to localStorage and the URL.

Technology choices

LayerToolWhy
Database Postgres 16 Real ACID transactions, JSONB, partial indexes, ENUM types, LATERAL joins. SQLite was the v1 draft; Postgres wins for queries and multi-host.
Migrations Flyway 10 OSS Versioned SQL files (V1__...sql) and repeatable migrations (R__...sql). Plain SQL, no DSL.
Container runtime Docker Compose Postgres, Flyway, and the Watcher all in one declarative file. Team workspaces bind-mounted read-only.
Watcher Python 3.12 + psycopg 3 Single-file script. psycopg 3 supports server-side prepared statements and is the maintained client.
Outbox transport Append-only JSONL Trivial to read, trivial to debug. No serialization library, no schema registry.
Cursor storage Per-role byte offset Resumable, idempotent on replay. A single integer per role per host.
File locking flock (Linux) / mkdir (macOS) Both atomic, both POSIX. Belt-and-braces for concurrent daemon appends.
Dashboard backend stdlib http.server Zero dependencies beyond psycopg. ThreadingHTTPServer for parallel requests.
Dashboard frontend Vanilla JS + Chart.js v4 One file, one CDN load, no build step. Easy to read, easy to fork.

Local vs central-server mode

Today the watcher and Postgres run on the same laptop as the daemons. Nothing in the schema requires that — the V3 migration added host_id and instance_id columns specifically so multiple machines can write to the same Postgres without colliding on the partial unique index. Moving to central mode is "change DATABASE_URL, run the daemons on more boxes". The schema is the same.
