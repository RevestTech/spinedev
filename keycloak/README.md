# `keycloak/` — Spine v3 embedded identity provider

> **What this is.** The Keycloak container and configuration that Spine Hub uses as its OIDC
> provider. Per locked decision #25, Keycloak is the **default identity provider for every Spine
> deployment, every tier, every shape (laptop / BYOC / customer-cloud / on-prem / air-gapped)**.
> Locked. Non-negotiable.
>
> **What this is NOT.** This subsystem does not perform application-layer auth itself — it only
> runs Keycloak and configures its realm. Spine Hub authenticates users by talking OIDC to the
> container in this subsystem via `shared/identity/keycloak_client.py` (Agent C's scope).

## Cross-agent contracts

These are the surfaces the rest of Spine talks to. **The values below are stable; if you change
them, coordinate with Agents A, B, C, D.**

| Contract | Value | Consumer |
|---|---|---|
| Issuer URL (laptop) | `http://localhost:8080/realms/spine` | `shared/identity/keycloak_client.py` (Agent C) |
| Issuer URL (production) | `https://${HUB_BASE_HOSTNAME}/auth/realms/spine` | same |
| Admin UI | `http://localhost:8080/admin` (laptop) / `https://${HUB_BASE_HOSTNAME}/auth/admin` (prod) | Human operators, Hub UI integration panel |
| Realm name | `spine` | `shared/identity/`, license bundle validation, Hub middleware |
| OIDC client_id | `spine-hub` | Hub SPA login flow, `shared/identity/middleware.py` |
| OIDC discovery | `${ISSUER_URL}/.well-known/openid-configuration` | Standard OIDC; SPA fetches once per session |
| JWKS endpoint | `${ISSUER_URL}/protocol/openid-connect/certs` | All token verifiers across Spine |
| Client secret vault path (default) | `secret/spine/keycloak/spine-hub/client-secret` | `shared/secrets/` (Agent C) reads; `vault/` (Agent D) writes |
| Management/health port | `9000` | `recovery/health.py` (Wave 5), monitoring |
| Default groups | `hub-admins`, `project-admins`, `developers`, `viewers`, `service-accounts` | RBAC in `shared/identity/rbac.py` |
| Default realm roles | `hub-admin`, `project-admin`, `developer`, `viewer`, `service-account` | RBAC in `shared/identity/rbac.py` |
| Token claims | `groups[]`, `realm_access.roles[]`, `hub_id`, plus standard OIDC `sub`/`email`/`name` | Hub middleware, audit-record actor attribution |

**What this subsystem does NOT touch (strict boundaries):**

- `shared/identity/` — Agent C's package; contract = OIDC standards only.
- `vault/` — Agent D's container; we emit vault path *references*, never write secrets there.
- Root `Makefile` — not modified by this subsystem.
- `shared/secrets/` — we *describe* the vault path; we don't implement secret resolution.

## First-time bootstrap flow

For a laptop deployment (the only shape this subsystem ships compose for):

```bash
# 1. Make sure the docker daemon is running.

# 2. Create an untracked .env.local with at least POSTGRES_PASSWORD set.
#    KEYCLOAK_ADMIN_PASSWORD can be set OR generated by --generate-admin in step 4.
cat > .env.local <<'EOF'
POSTGRES_PASSWORD=$(python3 -c 'import secrets,string; print("".join(secrets.choice(string.ascii_letters+string.digits) for _ in range(32)))')
KEYCLOAK_ADMIN_PASSWORD=changeme-then-rotate-immediately
EOF

# 3. Bring up Keycloak + its Postgres.
docker compose --env-file .env.local up -d

# 4. Run the Day-0 bootstrap. This:
#    - waits for /health/ready
#    - creates realm 'spine'  (idempotent: update if exists, preserve users)
#    - creates client 'spine-hub' with your Hub base URL
#    - seeds default groups + role mappings
#    - rotates client secret + emits vault-reference manifest
#    - (optional --generate-admin) rotates admin password to 32-char random
./init-bootstrap.sh \
  --keycloak-url http://localhost:8080 \
  --hub-base-url http://localhost:8090 \
  --admin-user admin \
  --admin-pass "$(grep KEYCLOAK_ADMIN_PASSWORD .env.local | cut -d= -f2)" \
  --generate-admin \
  --output bootstrap-credentials.txt

# 5. Move bootstrap-credentials.txt into your password manager.
#    Move the emitted client secret into vault at the path printed by the script.
#    THEN delete both files.
```

For BYOC / customer-cloud / on-prem / air-gapped: the Hub installer wraps `init-bootstrap.sh` and
substitutes vault calls for the file emission paths. The `docker-compose.yml` here is **laptop
only**; production deployments use Kubernetes manifests built in Wave 3.

## How `shared/identity/keycloak_client.py` (Agent C) talks to this container

1. **Discovery.** Agent C's client fetches `${ISSUER_URL}/.well-known/openid-configuration` once
   per process; caches with TTL = 1h or until JWKS rotation event.
2. **JWKS.** Periodic refresh of `${JWKS_URI}` every 10 minutes; immediate refresh on `kid` miss.
3. **Login redirect.** Hub SPA initiates code+PKCE flow → user authenticates at Keycloak (possibly
   through brokered IdP per `idp-presets/*`) → Keycloak redirects to Hub callback with `code` →
   Hub BFF exchanges `code` + `code_verifier` + `client_secret` (read from vault by Agent C) for
   tokens.
4. **Token validation.** Standard JWT verification against JWKS. RBAC done by reading `groups` +
   `realm_access.roles` claims (see contract table above) against feature flags from
   `tier-config.md`.

There is **no Keycloak-specific code in Hub** — only OIDC-standard code. That's the contract.

## Auth flow gotchas worth flagging to Wave 3 (Hub SPA OIDC login)

1. **PKCE is mandatory.** `spine-hub-client.json` sets `pkce.code.challenge.method=S256`. If the
   SPA omits PKCE, Keycloak rejects with a generic 400. Watch for this in dev.
2. **Refresh-token rotation is one-time-use.** `realm-config/spine-realm.json` sets
   `revokeRefreshToken=true` and `refreshTokenMaxReuse=0`. The SPA MUST replace its stored refresh
   token on every refresh; double-use of an old one revokes the session immediately. This is the
   security default but it breaks any "open multiple tabs and refresh both" patterns unless the
   SPA broadcasts the new token across tabs (use `BroadcastChannel`).
3. **`web-origins` must include the Hub origin exactly.** CORS preflights from the Hub SPA to
   Keycloak `/protocol/openid-connect/token` fail with no error in the SPA if origins mismatch.
   Pass `--add-web-origin` to `init-bootstrap.sh` for any non-default Hub origin (BYOC etc.).
4. **Frontchannel logout is on; backchannel logout is also on with `session.required=true`.** The
   SPA must implement the front-channel logout iframe handler OR the user's other tabs won't sign
   out cleanly.
5. **Group claim is non-full-path.** Token claim `groups` contains `["hub-admins", "developers"]`
   NOT `["/hub-admins", "/developers"]`. RBAC in `shared/identity/rbac.py` must strip the leading
   slash if it ever compares against the path.
6. **First-broker-login mode is `on`.** When a user logs in via Okta/Azure-AD/etc. for the first
   time, they hit a Keycloak "review profile" screen unless `firstBrokerLoginFlowAlias` is
   customised. For zero-friction enterprise SSO this should be flipped to a custom flow that
   auto-creates the user without the review step — Wave 3 to decide per tier.
7. **Email verification is required.** A user logging in via a brokered IdP that doesn't assert
   `email_verified=true` (rare but possible) will be blocked at first login. Document this in
   Wave 3 onboarding UX.
8. **Admin UI is at `/admin`, not `/auth/admin`.** Keycloak 18+ dropped the `/auth` prefix. If a
   reverse proxy expects the old layout, fix the proxy, not Keycloak.
9. **The bootstrap script regenerates client_secret on every run.** This is intentional (per #25
   "rotate often") but means Wave 3 should NEVER cache the secret outside vault; always re-read.
10. **Healthcheck uses port 9000.** Some K8s ingress controllers don't proxy the management port
    by default. Wave 3 K8s manifests must expose `9000` to the cluster-internal probe network.

## Operational quick-reference

| Task | Command |
|---|---|
| Start (laptop) | `docker compose --env-file .env.local up -d` |
| Stop | `docker compose down` |
| Stop + wipe data (DESTRUCTIVE) | `docker compose down -v` |
| Re-run Day-0 (idempotent) | `./init-bootstrap.sh ...` |
| Smoke test | `bash tests/test-keycloak-up.sh` |
| Tail logs | `docker logs -f spine-keycloak` |
| Open admin UI | `open http://localhost:8080/admin` (login as `admin` / password from bootstrap output) |
| Export realm (backup) | `docker exec spine-keycloak /opt/keycloak/bin/kc.sh export --dir /tmp/export --realm spine` |

For recovery, see `dr-runbook.md`.
For per-tier feature gating, see `tier-config.md`.
